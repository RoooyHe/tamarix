# Tamarix — 开源完备性评估

## 评估结论

项目代码架构扎实（47+ 组件、35 个 Matrix 模块、12 个 Store、i18n 双语），但缺少标准开源项目必需的基础设施。以下是按优先级分类的差距清单。

---

## P0 — 必须补齐（开源项目基本门槛）

### 1. 测试体系 — 完全缺失
- **现状**：零测试文件、零测试框架依赖、`package.json` 无 `test` 脚本
- **需要**：
  - Vitest 单元测试（`task-cache.ts`、`task-writes.ts`、`workflow.ts` 等纯逻辑层优先）
  - `@testing-library/svelte` 组件测试（关键业务组件如 TaskCard、KanbanBoard）
  - Playwright E2E 测试（核心流程：登录 → 创建任务 → 状态流转 → 看板拖拽）
  - `app.d.ts` 中取消注释 `App.Error` 等类型，补全错误边界类型定义

### 2. Lint & Format — 完全缺失
- **现状**：无 ESLint、无 Prettier、无 EditorConfig、无 `lint`/`format` 脚本
- **需要**：
  - ESLint（推荐 `eslint-plugin-svelte` + `typescript-eslint`）
  - Prettier（`prettier-plugin-svelte`）
  - `.editorconfig`（统一编辑器行为）
  - `package.json` 添加 `lint`、`format`、`lint:fix` 脚本

### 3. CI/CD — 完全缺失
- **现状**：无 `.github/` 目录、无任何 CI 配置
- **需要**：
  - `.github/workflows/ci.yml`：PR/push 触发 `typecheck` → `lint` → `test` → `build`
  - 可选：`.github/workflows/deploy.yml`（Vercel/Netlify 部署）

### 4. 社区文件 — 缺失
- **缺失**：`CONTRIBUTING.md`、`CODE_OF_CONDUCT.md`
- **需要**：
  - `CONTRIBUTING.md`：开发环境搭建、PR 规范、commit message 约定（Conventional Commits）、代码风格要求
  - `CODE_OF_CONDUCT.md`：Contributor Covenant 或类似标准

### 5. Git Hooks — 缺失
- **现状**：无 husky、无 lint-staged
- **需要**：
  - `husky` + `lint-staged`：提交前自动运行 lint + format

---

## P1 — 建议补齐（提升项目质量）

### 6. 错误边界 — 缺失
- **现状**：无任何 `+error.svelte` 页面，`App.Error` 类型被注释
- **需要**：
  - `src/routes/+error.svelte` — 全局错误回退页
  - 关键路由的 `+error.svelte`（如 `/project/[id]`、`/dashboard`）
  - 取消 `app.d.ts` 中 `App.Error` 的注释

### 7. 环境变量模板 — 根目录缺失
- **现状**：根目录无 `.env.example`（仅 `as/` 有）
- **需要**：
  - `.env.example` 列出前端所需的环境变量（如 `PUBLIC_MATRIX_HOMESERVER`）

### 8. CHANGELOG.md — 缺失
- **现状**：Release notes 分散在 `docs/release/` 目录
- **需要**：
  - `CHANGELOG.md` 聚合所有版本变更（或确认 `docs/release/` 为唯一 source of truth 并在 README 中链接）

### 9. docs/adr/ — 缺失
- **现状**：`docs/agents/domain.md` 引用了 `docs/adr/` 但该目录不存在
- **需要**：补充 ADR（Architecture Decision Records）或更新 domain.md 移除引用

### 10. 锁文件重复
- **现状**：`bun.lock` + `package-lock.json` 同时存在
- **需要**：删除 `package-lock.json`，统一使用 bun

### 11. 清理 artifact 文件
- **现状**：
  - `src/lib/components/dashboard/VersionProgressChart.svelte{}`
  - `src/lib/utils/crypto.ts{}`
- **需要**：删除这些带 `{}` 后缀的异常文件

## P2 — 可选增强

### 12. Storybook — 缺失
- 组件开发/文档化利器，尤其适合 32 个 shadcn-svelte 组件的展示
- 推荐：`@storybook/sveltekit`

### 13. 部署配置 — 缺失
- Dockerfile / docker-compose
- 或 Vercel / Netlify / Cloudflare Pages 配置文件

### 14. 自动化 a11y 测试
- 当前 a11y 依赖 shadcn-svelte 组件库（尚可），但业务组件无自动化检测
- 推荐：`@axe-core/playwright` 在 E2E 中集成

### 15. SEO 增强（低优先级，SPA 应用）
- 动态 `<title>` / `<meta description>` per route
- `sitemap.xml`（如需公开访问特定页面）

### 16. i18n 增强
- 当前手写 i18n 系统功能可用，但无复数化、无 ICU message format
- 可考虑迁移到 `paraglide-svelte` 或 `svelte-i18n`（非紧急）
